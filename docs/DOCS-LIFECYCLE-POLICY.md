# Documentation Lifecycle Management Policy

**Effective:** 2026-07-21  
**Scope:** All documentation in the BeckCloud repository

---

## 1. Purpose

Ensure documentation stays accurate, discoverable, and useful throughout the lifecycle of services and infrastructure.

---

## 2. Documentation Categories & Lifecycles

### Reference Docs (`docs/reference/`)

Auto-generated by Nova (AI sysadmin) from live cluster data. These are the source of truth for current state.

| Attribute | Value |
|-----------|-------|
| Author | Nova (auto-generated) |
| Review cadence | Periodic auto-update or on significant changes |
| Reviewer | Stephen (human) before committing |
| Retention | Current version only — superseded versions in git history |
| Metadata | Must include `Last audited:` date |

### Runbooks (`docs/runbooks/`)

Procedural guides for operational tasks.

| Attribute | Value |
|-----------|-------|
| Author | Stephen + Nova |
| Review cadence | After every deployment or configuration change |
| Reviewer | Stephen |
| Retention | Active runbooks kept current. Deprecated runbooks archived after 90 days |
| Metadata | Must include `Last updated:` date |

### Setup Guides (`docs/`)

Step-by-step procedural guides for initial configuration.

| Attribute | Value |
|-----------|-------|
| Author | Stephen + Nova |
| Review cadence | When the service/version changes |
| Retention | Kept indefinitely (historical setups may still be relevant) |

### Brand Docs (`docs/brand/`)

Brand identity, colors, logo specs, and website source.

| Attribute | Value |
|-----------|-------|
| Author | Stephen |
| Review cadence | On brand changes only |
| Retention | Current + archived versions in git |

### Maintenance Docs (`docs/maintenance/`)

SOPs for ongoing maintenance, rotation schedules, and recurring tasks.

| Attribute | Value |
|-----------|-------|
| Author | Stephen + Nova |
| Review cadence | Quarterly or after incidents |
| Retention | Active SOPs current. Superseded SOPs archived |

### Ansible Docs (`docs/ansible/`)

Procedural guides specific to Ansible provisioning.

| Attribute | Value |
|-----------|-------|
| Author | Stephen + Nova |
| Review cadence | After playbook changes |
| Retention | Current only |

### Archive (`docs/archive/`)

Completed plans, removed code, superseded docs.

| Attribute | Value |
|-----------|-------|
| Entry criteria | Work completed, code removed, or doc superseded |
| Retention | Indefinite — git history is the primary record, archive provides context |
| Cleanup | Archive items older than 2 years may be pruned at Stephen's discretion |

---

## 3. Update Triggers

Documentation must be updated when:

1. **Service deployed** — Add to namespace descriptions, services catalog, exposure map
2. **Service removed/disabled** — Document removal, update inventories
3. **Version upgraded** — Update HelmRelease versions, image tags, chart versions
4. **Configuration changed** — SSO chains, middleware, DNS, storage
5. **Infrastructure changed** — New namespace, PV/PVC, node added/removed
6. **Incident resolved** — Update runbook with lessons learned
7. **Playbook changed** — Update deployment runbook and relevant ansible docs

---

## 4. Naming Conventions

- **Filenames:** kebab-case, descriptive, prefixed with date if time-sensitive (`2026-07-21-topic.md`)
- **Headers:** H1 for title only, H2 for major sections, H3 for subsections
- **Cross-references:** Relative paths only (`[link](../reference/system-overview.md)`)
- **Commit messages:** `docs: <what> — <brief description>`

---

## 5. Review & Quality Gates

- Every doc file must have a `Last updated:` or `Last audited:` date at the top
- No credentials, secrets, or plaintext keys in any doc file
- No "TODO" or "FIXME" without a corresponding GitHub issue
- Auto-generated docs must be human-reviewed before committing
- Stale docs (older than 6 months without update) flagged for review

---

## 6. Responsibilities

| Role | Responsibility |
|------|---------------|
| Stephen | Final review, brand docs, setup guides, policy updates |
| Nova | Auto-generation of reference docs, initial drafts, data collection |
| Anyone | Reporting outdated docs via GitHub issues |
